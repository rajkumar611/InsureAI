    ```python
    # Create ground truth: for each customer, mark which claims are relevant

            ("CLM-2023-001", True),   # water damage — relevant
            ("CLM-2023-002", False),  # motor claim — not relevant
        ]
    }

    # Test retrieval
    retrieved = rag_search(customer_id="CUST-NZ-001", limit=5)

    # Measure quality
    precision = count_relevant_in_top_k(retrieved, ground_truth) / k
    recall = count_relevant_in_top_k(retrieved, ground_truth) / total_relevant
    mrr = 1.0 / (rank_of_first_relevant + 1)
    ndcg = calculate_ndcg(retrieved, ground_truth)
    ```

[ ] 2.1 — Advanced RAG: Query Expansion & Reranking
    File: backend/src/underwriting/pipeline_agents/claims_history_agent/rag_advanced.py

    Current approach: Exact → Fuzzy → Vector (3-tier fallback)
    Problem: Retrieves by similarity, not by relevance to decision

    New approach:
    ```python
    async def advanced_rag_search(query: str, customer_id: str) -> List[Claim]:
        # 1. Query expansion: rewrite query to catch more variants
        expanded_queries = await rewrite_query(query)
        # e.g., "water damage" → ["water damage", "flooding", "inundation", "water loss"]

        # 2. Dual retrieval: BM25 (exact match) + semantic (meaning)
        bm25_results = bm25_search(expanded_queries, customer_id)
        semantic_results = pgvector_search(query_embedding, customer_id)

        # 3. Combine and deduplicate
        candidates = merge_results(bm25_results, semantic_results)

        # 4. Rerank by decision relevance
        reranked = rerank(
            candidates=candidates,
            query=query,
            context=submission_context,  # what decision are we making?
            metric="relevance_to_risk_decision"
        )

        # 5. Evaluate quality
        metrics = {
            "precision@3": evaluate_precision(reranked[:3]),
            "mrr": calculate_mrr(reranked),
            "ndcg": calculate_ndcg(reranked),
        }

        return reranked[:5]
    ```

    Why: Advanced RAG is table-stakes for senior AI roles.
         Shows understanding of IR beyond vector search.

[ ] 2.3 — Advanced Error Handling & Recovery
    File: backend/src/underwriting/platform/resilience.py

    Current: Basic try/catch
    Problem: Single point of failure crashes pipeline

    New patterns:
    ```python
    # 1. Graceful degradation
    async def get_risk_assessment(state: WorkflowState) -> RiskDecision:
        try:
            return await sonnet_risk_assessment(state)
        except RateLimitError:
            logger.warning("Sonnet rate limited, falling back to Haiku")
            return await haiku_risk_assessment(state)  # Cheaper, simpler
        except Exception as e:
            logger.error(f"Risk assessment failed: {e}")
            return await rules_based_decision(state)  # Pure deterministic

    # 2. Retry with exponential backoff
    @retry(
        wait=wait_exponential(multiplier=1, min=4, max=10),
        stop=stop_after_attempt(3),
        reraise=True
    )
    async def call_llm_with_retry(model, prompt):
        return await anthropic_client.messages.create(...)

    # 3. Circuit breaker (stop hammering failing service)
    breaker = CircuitBreaker(
        failure_threshold=5,
        recovery_timeout=60
    )

    async def call_external_api():
        async with breaker:
            return await external_service.request()

    # 4. Fallback chain
    decision = (
        await try_sonnet(state)
        or await try_haiku(state)  # Fallback to cheaper model
        or await rules_based_decision(state)  # Fallback to rules
    )
    ```

    Why: Production systems must handle failures gracefully.
         Shows understanding of reliability engineering.

[ ] 2.4 — Fairness & Bias Analysis
    File: backend/tests/evals/fairness_analysis.py

    Purpose: Audit for discrimination in insurance decisions

    Tests:
    ```python
    # 1. Disparate impact analysis
    # Are we declining certain demographics at different rates?

    demographic_groups = {
        "north_island": [submission for s in test_set if s.jurisdiction == "NZ" and s.location_north],
        "south_island": [submission for s in test_set if s.jurisdiction == "NZ" and s.location_south],
    }

    decline_rate_north = sum(1 for s in demographic_groups["north_island"] if s.decision == "DECLINE") / len(demographic_groups["north_island"])
    decline_rate_south = sum(1 for s in demographic_groups["south_island"] if s.decision == "DECLINE") / len(demographic_groups["south_island"])

    ratio = max(decline_rate_north, decline_rate_south) / min(decline_rate_north, decline_rate_south)

    # If ratio > 1.25, we have disparate impact (under 4/5 rule)
    if ratio > 1.25:
        print(f"WARNING: Disparate impact detected (ratio: {ratio:.2f})")
        print(f"North Island decline rate: {decline_rate_north:.1%}")
        print(f"South Island decline rate: {decline_rate_south:.1%}")

    # 2. Fairness metrics
    # Given same risk profile, do we decide same way?

    matched_pairs = find_similar_submissions(test_set)
    for pair in matched_pairs:
        if pair[0].decision != pair[1].decision:
            print(f"Different decisions for similar risk profiles: {pair[0].id} vs {pair[1].id}")

    # 3. Consistency audits
    # Run same submission twice, do we get same decision?

    for submission in test_set:
        decision1 = await pipeline.run(submission)
        decision2 = await pipeline.run(submission)

        if decision1 != decision2:
            print(f"Non-deterministic decision for {submission.id}")
    ```

    Report: evals/fairness_audit_report.md

    Why: Insurance is heavily regulated on fairness (PDPO in HK).
